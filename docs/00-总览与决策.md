## 一、一句话定义这个项目

构建 **`astrbot_plugin_everos`**：一个把 EverOS 自进化记忆引擎接入 AstrBot 的插件，
通过 `on_llm_request` / `on_llm_response` 钩子**自动**完成「每轮注入用户画像+情景记忆」和「对话归档」，
并按平台真实身份（QQ 号）做**确定性硬隔离**。

它要同时解决两个层面的问题：
1. **记忆架构层**（EverOS 引擎负责）：持久用户画像、按 owner_id 硬隔离、聚类去重——解决「印象漂移 / 跨用户串线 / 同质过饱和」。
2. **集成层**（本插件负责）：把 EverOS 的能力真正接进 AstrBot 的对话流，且不重蹈现有集成插件的覆辙。

## 二、为什么是 B 方案（三选一的结论）

在三条路径里做过权衡：

| 方案 | 描述 | 结论 |
|---|---|---|
| A. 自建薄中间件 | 留在 Mnemosyne，自己加持久画像+硬过滤+visibility | ❌ 不选。「持久画像的提炼/巩固算法」是最难调的部分，自己从零写不划算 |
| **B. EverOS 引擎 + 自写集成层** | 用 EverOS 做记忆底座，抛开半成品插件，自己写干净的 AstrBot 接入 | ✅ **选这个** |
| C. 修现有集成插件 | 在 `astrbot_plugin_everos_integration` 上补钩子 | ❌ 不选。它 README 与代码不符、身份映射有设计缺陷，排查成本可能高于重写 |

**B 方案的核心论据（全部源码实证，详见 `01-实证依据.md`）：**

- EverOS 引擎确实把最难的两件事做好了：
  - **持久用户画像**：`users/<user_id>/user.md`，LLM 增量 UPDATE，按 owner_id KV 直取、不靠相似度召回。
  - **四重硬隔离**：`owner_id / owner_type / app_id / project_id` 强制 AND 进每条查询，DSL 不可覆盖。
- 现成集成插件不能直接用：
  - **没有 `on_llm_request` 注入钩子**——它只是个被动工具库，不会自动把记忆喂给 LLM。
  - **身份映射有缺陷**——`user_id` 靠 LLM 传参，默认落 `"default"`，检索还轮询 `[uid, "default"]`，等于把所有用户汇进公共池，**反而放大跨用户串线**。
  - **README 与代码不符**——宣称的 `retrieval_hook.py` / `dialog_sync.py` 在代码里不存在。
- Mnemosyne 是成熟的正面参考：它的 `on_llm_request` 注入范式、从 `event` 确定性取 sender_id 的做法，是 B 方案要复刻的正确姿势。

## 三、范围边界（务必先对齐，避免范围蔓延）

### 本插件要做的（MVP 必须）

1. **自动记忆注入**：`on_llm_request` 钩子里，按当前用户身份检索 EverOS（含 `include_profile=true`），把「用户画像 + 相关情景」拼进 `req.system_prompt`。
2. **自动对话归档**：`on_llm_response` 钩子里，把本轮 user+assistant 消息 `add` 进 EverOS，在合适时机 `flush`。
3. **确定性身份硬隔离**：从 `event` 取 QQ 号作 `user_id`（绝不靠 LLM 传参、绝不回退 `"default"`），群聊用 `group_id` 参与 scope 决策。
4. **EverOS 客户端**：复用现有 `everos_client.py` 的端点封装思路（已验证可用），补齐 `include_profile`、`filters`、`agent_id` 等本报告确认的参数。
5. **降级安全**：EverOS 不可达时，钩子静默跳过、绝不阻断对话（参考 Mnemosyne 的 try/except 包裹）。
6. **管理命令**：`/everos` 命令组（status / search / forget / help），管理员权限控制。
7. **配置 schema**：`_conf_schema.json`，含连接、注入开关、注入位置、隔离策略、归档触发策略。
8. **随附 Docker 部署**：仓库带 `deploy/`（自建 Dockerfile + docker-compose + .env 模板），让 EverOS 与 AstrBot 同处一个 docker 网络，用户一次 `docker compose up` 即可。详见 `06-Docker部署与开箱即用.md`。

### 本插件可选做的（v1.1+，先留接口）

- visibility 隐私分层（PUBLIC/PRIVATE）的注入侧过滤——EverOS schema 不原生支持自定义标签，需在注入层用旁路策略实现。详见 `02-架构设计.md`。
- WebUI 管理面板（现有插件的 standalone_server 思路可参考，但非 MVP）。
- 群聊多人记忆的精细归属。

### 本插件明确不做的

- ❌ **不在插件进程内嵌 EverOS 引擎**——这是 EverOS 的硬约束（`QUICKSTART.md` 明文「无 in-process library mode，everos server 必须独立运行在 agent 前面」），不是保守选择。开箱即用靠**随附 compose 把 EverOS 容器化**实现，不靠内嵌。
- ❌ **插件不调用 docker 命令、不自动拉起容器**——用户 AstrBot 在容器里，插件调 docker 要挂 socket、有安全代价；用户已确认「一次手动 up 可接受」。插件只做纯 HTTP 客户端。
- ❌ 不实现记忆的「遗忘/衰减」（EverOS 引擎本身不提供，超出集成层职责）。
- ❌ 不改 EverOS 源码（只通过其公开 HTTP API 交互）。
- ❌ 不做跨平台身份统一（QQ 以外平台的身份映射留作后续扩展点）。

## 四、验收标准（这份报告交付的代码要能通过）

构建完成后，新会话应能自证以下各点（详细测试集见 `03-构建任务清单.md`）：

1. **注入生效**：开启插件后，发一条消息，能在 EverOS 服务日志/插件 debug 日志里看到一次 `search` 调用，且 `req.system_prompt` 被注入了记忆段。
2. **身份硬隔离**：用户 A（QQ 号 a）的检索请求，传给 EverOS 的 `user_id` 严格等于 a，绝不出现 `"default"` 或轮询多 uid。代码层面 grep 不到 `"default"` 兜底。
3. **归档生效**：一轮对话后，EverOS 里能 `get` 到该 user_id 名下新增的 episode。
4. **降级安全**：停掉 EverOS 服务，对话仍正常进行，插件只在日志告警，不抛异常给用户。
5. **可审计**：核心模块有清晰分层，无中文键名混英文、无 README 与代码不符、关键决策有注释说明出处。
6. **可安装**：符合 AstrBot 插件规范（metadata.yaml / _conf_schema.json / requirements.txt 齐全），能被 AstrBot 识别加载。
7. **可开箱部署**：`deploy/` 下 compose 能起 EverOS 容器，与 AstrBot 同网络；插件默认 `everos_base_url` 指向同网络服务名即可连通；`/everos status` 能正确反映连接状态。
