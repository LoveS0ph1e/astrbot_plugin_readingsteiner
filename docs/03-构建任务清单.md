> 新会话照此顺序执行。每个任务有「产出」和「验收点」。**前置任务 0 是迁移前的现实检查，强烈建议先做。**
> 标记：⭐ = MVP 必须；🔹 = 可延后到 v1.1；⚠️ = 易错点，对照 `01-实证依据.md`。

## 任务 0 ⭐（前置·验证而非编码）：实测 EverOS 提取质量

**为什么**：B 方案的全部价值押在「EverOS 的持久画像比 Mnemosyne 的向量片段强」。这必须先用真实数据验证，否则可能白迁移。

**做什么**：
1. 按 `EverOS/QUICKSTART.md` 起一个本地 EverOS 服务（`ENV=DEV everos server start`，开 `/docs`）。
2. 用 `curl` 或写个小脚本，把一段你真实风格的多轮对话（可用茕霞线脱敏样本）`POST /add` 进去，再 `/flush`。
3. `POST /search` 带 `include_profile=true`，看返回的 `profiles[].profile_data` 长什么样。
4. **判断**：它提炼出的用户画像，能不能稳定承载「深羁绊 + 多面向」的复杂用户印象？比 Mnemosyne 每轮重新推断强多少？

**验收点**：能回答「EverOS 的画像质量是否值得迁移」。若否，停止，回到方案 A。
**产出**：一份简短的实测记录（贴进本目录 `experiments/everos-profile-probe.md`）。

## 任务 1 ⭐：脚手架与插件骨架

**做什么**：建 `02-架构设计.md` 第一节的目录结构。先建空文件 + 三个元文件。

`metadata.yaml`（字段照 AstrBot 官方模板 `astrbot-plugin-template/metadata.yaml` 实证；`name` 须 `astrbot_plugin_` 前缀、全小写、无空格——官方 `plugin-new.md:21-25`）：
```yaml
name: astrbot_plugin_everos              # 唯一识别名，astrbot_plugin_ 前缀，全小写
display_name: EverOS 记忆                 # 展示名（需 AstrBot >= v4.5.0，低版本忽略不报错）
desc: 基于 EverOS 自进化记忆引擎的 AstrBot 长期记忆插件（持久用户画像 + 按身份硬隔离）
help: 输入 /everos help 查看指令
version: v0.1.0
author: <你的名字>
repo: <你的仓库URL>
```

`requirements.txt`：
```
httpx>=0.27.0
```
（fastapi/uvicorn 仅 WebUI 才需要，MVP 不做 WebUI，不加）

`main.py` 骨架：`@register(...)` + 空的三个钩子 + terminate + 命令组占位。导入照 `01-实证依据.md` 3.1。

**验收点**：AstrBot 能加载插件、识别 `/everos` 命令组、日志无报错。

## 任务 1.5 ⭐：Docker 编排（deploy/，开箱即用的基础）

**做什么**：照 `06-Docker部署与开箱即用.md` 第三节，建 `deploy/` 下四个文件：
- `Dockerfile`：`python:3.12-slim` + `pip install everos==1.0.1`，设 `EVEROS_API__HOST=0.0.0.0`、`EVEROS_MEMORY__ROOT=/data/everos`，`CMD ["everos","server","start"]`。
- `docker-compose.yml`：everos 服务 + `everos_data` volume + healthcheck(`/health`) + `astrbot_net` 网络。
- `.env.example`：四个 API key 槽（两个 key），含 DashScope 备选注释。
- `deploy/README.md`：三步部署说明。

⚠️ 关键点（都在 06 实证）：容器内必须绑 `0.0.0.0`；数据挂 volume 持久化；与 AstrBot 同 docker 网络，插件用服务名 `http://everos:8000` 连。

**验收点**：`docker compose up -d` 能起 everos 容器，`curl http://127.0.0.1:8000/health` 返回 ok；重启容器记忆不丢（volume 生效）。

## 任务 2 ⭐：EverOS 客户端（core/everos_client.py）

**做什么**：实现 HTTP 异步客户端。可基于 `astrbot_plugin_everos_integration/core/everos_client.py` 改写，但必须：

- ⚠️ base_url 默认 `http://127.0.0.1:8000`（不是现有插件的 8765），从配置读。
- 实现方法：`health()` / `add()` / `flush()` / `search()` / `get()`。
- ⚠️ `search()` 必须支持本报告确认的全部关键参数：`user_id`, `query`, `app_id`, `project_id`, `method`, `top_k`, `radius`, `include_profile`, `filters`, `agent_id`。对照 `01-实证依据.md` 2.4。
- ⚠️ 所有响应先取 `data` 字段（响应包络，`01` 2.1）；错误响应取 `error`。
- ⚠️ `get()` 的 memory_type 只用 `episode`/`profile`/`agent_case`/`agent_skill`（不是 `user_profile`，`01` 2.5）。
- 统一异常处理：网络错误抛自定义 `EverOSUnavailable`，让上层 try/except 捕获降级。
- `close()` 方法关闭 httpx client，供 terminate 调用。

**验收点**：`tests/test_everos_client.py` 用 mock httpx 覆盖每个方法的请求体构造与响应解析。对照 api.md 的 cURL 样例断言请求体字段。

## 任务 3 ⭐：身份解析（core/identity.py）

**做什么**：实现 `resolve(event) -> IdentityResult | None`，把 event 映射成 `(user_id, session_id, app_id, project_id, is_group, bot_id)`。

- ⚠️ user_id = 发送者真实 QQ 号，从 event 确定性取（参考 `Mnemosyne/core/tools.py` 的 sender 解析、`memory_operations.py:186-190` 的 `_resolve_sender_identity`）。
- session_id = `event.unified_msg_origin`。
- is_group = `event.get_group_id() != ""`（`tools.py:346-350`）。
- app_id：按 `isolation_personas` 白名单决定是否加人格后缀（参考 `config_manager.get_app_id_for`）。
- ⚠️ **铁律**：取不到 user_id 返回 None；绝不 default 兜底；绝不轮询多 uid。把 `02-架构设计.md` 4.3 的三条铁律写成模块顶部注释。

**验收点**：`tests/test_identity.py` 必须覆盖——
1. 私聊：user_id = 对方 QQ。
2. 群聊：user_id = 发言者 QQ（非群号），session 含群标识。
3. 取不到身份：返回 None。
4. 隔离回归测试：两个不同 QQ 号解析出的 user_id 不相等、不为 default。（这是防 §15.9 串线的回归防线）

## 任务 4 ⭐：注入逻辑（core/injection.py）

**做什么**：
1. `build_text(profiles, episodes, config) -> str`：把检索结果拼成 `02-架构设计.md` 3.1 的注入文本结构（画像段 + 情景段，带可配置前后缀标签）。
2. `inject(req, text, target, position)`：按配置改写 `req.system_prompt` 或 `req.prompt`，prepend/append。对照 `01-实证依据.md` 3.3 的 Mnemosyne 范式。
3. 处理边界：profiles 为空只注情景；都为空则不注入。

**验收点**：`tests/test_injection.py` 覆盖——画像+情景都有、只有画像、只有情景、都为空四种情况；注入到 system_prompt 与 user_prompt 两种 target；prepend/append 两种位置。断言 req 字段被正确改写。

## 任务 5 ⭐：归档逻辑（core/archiving.py）

**做什么**：
1. `build_messages(event, resp, identity) -> list[dict]`：构造 user+assistant 两条 MessageItem。
   - ⚠️ user 消息 sender_id = QQ 号，assistant 消息 sender_id = bot_id。
   - ⚠️ timestamp = `int(time.time() * 1000)`（毫秒，`01` 2.2）。
   - ⚠️ role 用 `"user"`/`"assistant"`。
2. `should_flush(session_id, strategy, idle_seconds) -> bool`：实现 auto/every_turn/manual 三策略（`02` 3.3）。
   - auto 策略需一个轻量的「会话最后活动时间」记录 + 定时检查（可用 asyncio 后台任务，参考 Mnemosyne 的 message_counter 思路）。

**验收点**：every_turn 策略下，一轮对话触发一次 add + 一次 flush；auto 策略下只 add，静默超时后才 flush。

## 任务 6 ⭐：主入口接线（main.py）

**做什么**：把 core 模块接到三个钩子上。每个钩子体严格遵循：

```python
@filter.on_llm_request()
async def on_llm_request(self, event, req):
    try:
        if not self._enabled(event):          # 开关/会话白名单检查
            return
        identity = identity_mod.resolve(event)
        if identity is None:                   # 铁律：无身份则跳过
            return
        if not await self._client_healthy():   # EverOS 不可达则跳过
            return
        results = await self.client.search(
            user_id=identity.user_id, query=...,
            include_profile=self.config.get("include_profile", True),
            top_k=..., app_id=identity.app_id, project_id=identity.project_id)
        if self.config.get("group_public_only") and identity.is_group:
            results = visibility.filter_public(results)   # 可选
        text = injection.build_text(results["profiles"], results["episodes"], self.config)
        injection.inject(req, text, target, position)
    except Exception as e:
        logger.error(f"[everos] on_llm_request 失败: {e}", exc_info=True)
    return
```

- ⚠️ 三个钩子全部 try/except 包裹（`01` 3.4）。
- `on_astrbot_loaded`（或 `initialize`，二者官方均确认可用）里初始化 client、起 auto-flush 后台任务。
- `terminate` 里关 client、取消后台任务。

**验收点**：见任务 8 的端到端验收。

## 任务 7 ⭐：管理命令（commands/handlers.py）

**做什么**：`/everos` 命令组：
- `/everos status`：显示 EverOS 健康、当前 scope、本会话 user_id、记忆计数（调 get/stats）。
- `/everos search <query>`：手动检索当前用户记忆，展示结果（调试用）。
- `/everos forget`：⚠️ 删除当前用户记忆——高危操作，需管理员权限 + 二次确认（参考 `Mnemosyne/main.py:659` 的 `@filter.permission_type(ADMIN)` + confirm 参数模式）。
- `/everos flush`：手动 flush 当前会话。
- `/everos help`：帮助。

**验收点**：非管理员调 forget 被拒；forget 需 confirm 参数。

## 任务 8 ⭐：端到端验证（对应 00 的验收标准）

逐条验证 `00-总览与决策.md` 第四节的 6 个验收标准：

| # | 测试 | 期望 |
|---|---|---|
| 1 | 起 EverOS，发一条消息 | 日志见一次 search，req.system_prompt 被注入 |
| 2 | 两个不同 QQ 号各发消息 | 传给 EverOS 的 user_id 各为其 QQ，grep 代码无 "default" 兜底 |
| 3 | 一轮对话后查 EverOS | get 到该 user_id 新增 episode |
| 4 | 停掉 EverOS 再发消息 | 对话正常，日志告警，无异常抛给用户 |
| 5 | 代码审查 | 分层清晰、无中文键混英文、README 与代码一致 |
| 6 | AstrBot 加载 | 插件被识别、命令可用 |

⚠️ 特别测：**串线回归**——用户 A 存「我喜欢猫」，用户 B（不同 QQ）问「我喜欢什么」，B 的检索绝不返回 A 的记忆。这是 §15.9 的根治验证。
