## 第一部分：EverOS 引擎能力（决定「值得接」）

### 1.1 持久用户画像 —— 解决「印象漂移」

EverOS 有「每用户一份、可增量更新、KV 直取」的持久画像，这正是 Mnemosyne 没有、而你审计文档 §15.10 反复要的东西。

| 事实 | 证据 |
|---|---|
| 画像存储为 `users/<user_id>/user.md`，含 summary / explicit_info / implicit_traits / profile_timestamp_ms | `EverOS/docs/storage_layout.md`；`src/everos/infra/persistence/markdown/mds/profile.py:20-40` |
| 画像由 LLM 接收「旧画像 + 新 MemCell 集合」做 INIT/UPDATE，不是简单 append | `src/everos/memory/strategies/extract_user_profile.py:151-156` |
| 画像按 owner_id 做 KV lookup 获取，**不经相似度召回** | `src/everos/memory/search/manager.py:456-461`；`infra/persistence/lancedb/tables/user_profile.py` |
| 新记忆先聚类、与既有 cluster 几何相似度 merge（非逐条堆叠） | `src/everos/memory/strategies/trigger_profile_clustering.py:42-60` |
| 内容用 content_sha256 去重，内容不变不重复索引 | `src/everos/memory/cascade/handlers/user_profile.py:84-92` |

**对集成层的含义**：检索时必须传 `include_profile=true` 才会返回画像（默认 false）。画像是「每轮该注入的稳定槽位」的天然载体。

### 1.2 四重硬隔离 —— 解决「跨用户串线」

| 事实 | 证据 |
|---|---|
| 每条查询强制 AND 上 `owner_id / owner_type / app_id / project_id` 四个条件 | `src/everos/memory/search/filters.py:100-121` |
| 这四个字段在 filter DSL 里是 **reserved，不允许覆盖**，只能在请求顶层设置 | `docs/api.md:393-399`；`filters.py:84-86` |
| 磁盘路径也按 scope 分区：`~/.everos/<app>/<project>/users/<user_id>/...` | `docs/api.md:136-147`；`docs/storage_layout.md` |
| `/search` / `/get` 查询永不跨 scope | `docs/api.md:146-147` |

**对集成层的含义**：只要插件把 QQ 号正确传成 `user_id`，EverOS 在引擎层就保证 A 用户绝不会召回 B 用户的记忆。**串线的唯一风险点被前移到了「插件有没有正确传 user_id」——这正是现有集成插件搞砸的地方（见 3.3）。**

### 1.3 EverOS 不做的事（集成层要心里有数）

| 事实 | 证据 | 影响 |
|---|---|---|
| 无「主动遗忘 / 衰减 / TTL」——老 episode 永不删 | 全文档无 forget/decay/expire 机制 | 过饱和只被「聚类去重」缓解一半；插件不补这个 |
| 不支持自定义元数据标签（如 visibility: public/private） | schema 无此字段；profile_data 是 LLM 生成的自由结构 | visibility 隐私分层要在**注入层**旁路实现，非 EverOS 原生 |
| 无内置鉴权，默认只绑 127.0.0.1 | `docs/api.md:55-60` | 插件连本地 EverOS 安全；若 EverOS 暴露公网需用户自行加网关 |
| 读写最终一致：flush 后 search 可能有 ~1s（高负载 10-15s）延迟 | `docs/api.md:79-90` | 归档后不要立即依赖能检索到；注入读的是历史已沉淀记忆，不受影响 |

## 第二部分：EverOS HTTP API 契约（新会话照这个调，不要脑补）

> 全部来自 `EverOS/docs/api.md` 精读。Base URL 默认 `http://127.0.0.1:8000`（文档），
> 但现有集成插件用的是 `8765`（`everos_client.py:21`）——**端口以用户部署配置为准，做成可配置项**。
> 所有业务端点在 `/api/v1/memory/` 下，**全部是 POST**（即使语义像读）。`docs/api.md:132-134`

### 2.1 响应包络（所有 200 响应统一结构）

```json
{ "request_id": "<32位hex>", "data": { /* 端点专属负载 */ } }
```
错误响应：`{ "request_id": "...", "error": { "code", "message", "timestamp", "path" } }`
证据：`docs/api.md:62-77, 159-194`。**客户端解析时一律先取 `data`。**

### 2.2 POST /api/v1/memory/add —— 写入（归档用）

请求体（`docs/api.md:474-495`）：

| 字段 | 类型 | 必填 | 默认 | 约束 |
|---|---|---|---|---|
| `session_id` | string | 是 | — | 长度 1–128 |
| `app_id` | string | 否 | `"default"` | `^[a-zA-Z0-9_.-]+$`，1–128 |
| `project_id` | string | 否 | `"default"` | 同上 |
| `messages` | array<MessageItem> | 是 | — | 1–500 条 |

MessageItem（`docs/api.md:203-211`）：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `sender_id` | string | 是 | **role=user 时这就是记忆索引的 user_id**。minLength=1 |
| `sender_name` | string\|null | 否 | 显示名，LLM 提取时可见，不作索引键 |
| `role` | `"user"\|"assistant"\|"tool"` | 是 | — |
| `timestamp` | integer | 是 | **Unix epoch 毫秒(ms)**，必须 > 0 |
| `content` | string \| array<ContentItem> | 是 | 纯文本直接给 string |

响应：`data.status` = `"accumulated"`（进缓冲，边界未触发）或 `"extracted"`（本次触发了提取）。`docs/api.md:498-515`

**关键**：`sender_id` 对 role=user 就是索引键。归档时 user 消息的 sender_id 必须填 QQ 号，assistant 消息填一个稳定的 bot 标识（如 `assistant` 或 self_id）。

### 2.3 POST /api/v1/memory/flush —— 强制提取（对话结束时）

请求体：`session_id`（必填）+ `app_id` + `project_id`。`docs/api.md:554-560`
响应：`data.status` = `"extracted"` 或 `"no_extraction"`（缓冲为空）。
**markdown 落盘是同步的**（返回即在磁盘），但 LanceDB 索引异步。`docs/api.md:581-584`

### 2.4 POST /api/v1/memory/search —— 检索（注入用，最重要）

请求体（`docs/api.md:614-626`）：

| 字段 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `user_id` | string\|null | **与 agent_id 二选一** | null | 设它就查 user track（episodes/profiles）。minLength=1 |
| `agent_id` | string\|null | 与 user_id 二选一 | null | 设它查 agent track |
| `app_id` | string | 否 | `"default"` | scope |
| `project_id` | string | 否 | `"default"` | scope |
| `query` | string | 是 | — | minLength=1 |
| `method` | `keyword\|vector\|hybrid\|agentic` | 否 | `"hybrid"` | hybrid 是推荐默认 |
| `top_k` | integer | 否 | `-1` | -1=服务端默认；或 1..100 |
| `radius` | number\|null | 否 | null | 余弦相似度阈值 [0,1]，砍弱匹配长尾 |
| `include_profile` | boolean | 否 | `false` | **设 true 才返回用户画像** |
| `enable_llm_rerank` | boolean | 否 | false | 加一次 LLM rerank |
| `filters` | FilterNode\|null | 否 | null | 过滤 DSL，见 2.6 |

**`user_id` / `agent_id` 必须恰好设一个**，否则 422。`docs/api.md:628-630`

响应 `data`（五个数组恒在，不适用的为 `[]`）：`docs/api.md:680-691`
- `episodes`：array<SearchEpisodeItem>，user_id 设时填充
- `profiles`：array<SearchProfileItem>，**user_id 设且 include_profile=true 时填充**
- `agent_cases` / `agent_skills`：agent_id 设时填充
- `unprocessed_messages`：仅当 filters 顶层有 session_id eq 时填充

SearchEpisodeItem 关键字段（`docs/api.md:699-713`）：`id / user_id / session_id / timestamp(ISO8601) / sender_ids / summary(~200字) / subject(一行) / episode(完整叙述) / score / atomic_facts[]`
SearchProfileItem（`docs/api.md:730-737`）：`id / user_id / app_id / project_id / profile_data(object,自由结构) / score(直取时为 null)`

**注入策略含义**：注入时取 `profiles[].profile_data`（稳定画像）+ `episodes[].summary` 或 `episode`（相关情景）。画像优先、恒注入；情景按 score 取 top 几条。

### 2.5 POST /api/v1/memory/get —— 分页列举（管理命令/统计用）

请求体：`user_id`|`agent_id`（二选一）+ `app_id` + `project_id` + `memory_type`（必填）+ `page` + `page_size`(1-100) + `sort_by`(`timestamp`|`updated_at`) + `sort_order`。`docs/api.md:860-871`
`memory_type` 取值（`docs/api.md:448-457`）：`episode`/`profile`（要 user_id）、`agent_case`/`agent_skill`（要 agent_id）。
响应含 `total_count`（分页前总数）+ `count`（本页数）。

> 注意：现有 `everos_client.py:122-144` 的 `memory_get` 注释说「memory_type 可取 user_profile / foresight」，但 `docs/api.md` 的 GetMemoryType 只列了 4 种（episode/profile/agent_case/agent_skill）。**以 api.md 为准**，新插件用 `profile` 不用 `user_profile`。这是现有插件可能的一处 bug，新会话别抄。

### 2.6 FilterNode 过滤 DSL（可选，visibility 旁路可能用到）

- 递归布尔树：`AND`/`OR` 数组 + 字段谓词，同级隐式 AND。`docs/api.md:352-376`
- **允许的字段**：`session_id / parent_type / parent_id / timestamp / sender_id`。其它字段 422。`docs/api.md:380-399`
- **reserved（不可入 DSL，只能顶层设）**：`owner_id / owner_type / app_id / project_id`。
- 操作符：`eq/ne/gt/gte/lt/lte/in`。`docs/api.md:404-416`
- `sender_id` 是数组成员匹配（`array_has`）。

### 2.7 健康检查

`GET /health`，现有客户端判 `data.status == "ok"`（`everos_client.py:30-41`）。`/health` 和 `/metrics` 在 `/api/v1` 之外。`docs/api.md:45-48`

### 3.3 记忆如何注入 ProviderRequest（B 方案核心，照抄这个模式）

Mnemosyne 在 `on_llm_request` 钩子里改写 `req`，支持三种注入方式 + 两种位置：

```python
# 证据：Mnemosyne/core/memory_operations.py:995-1046
# 1) 拼装记忆文本（带可配置的前后缀标签）
long_memory_prefix = config.get("long_memory_prefix", "<Memory> 长期记忆片段：")
long_memory_suffix = config.get("long_memory_suffix", "</Memory>")
long_memory = f"{long_memory_prefix}\n" + ...每条记忆... + long_memory_suffix

# 2) 按配置选注入方式与位置
injection_method = config.get(...)        # "user_prompt" / "system_prompt" / "insert_system_prompt"
injection_position = config.get("memory_injection_position", "prepend")  # prepend / append

# 注入到 user prompt：
req.prompt = long_memory + "\n" + req.prompt        # prepend
# 或注入到 system prompt：
req.system_prompt = long_memory + req.system_prompt  # prepend
```

**关键 API 事实（AstrBot 源码 `entities.py:89-114` 实证 ProviderRequest 完整字段）**：
`prompt: str|None` / `system_prompt: str` / `contexts: list[dict]`(OpenAI 格式历史) / `image_urls: list` / `audio_urls: list` / `extra_user_content_parts: list[ContentPart]`(在用户消息后追加内容块) / `func_tool` / `conversation` / `model` / `session_id`。
→ 注入可改的字段：`system_prompt`(推荐)、`prompt`、`extra_user_content_parts`(官方文档 `plugin.md:511-549` 推荐的追加方式)。
**LLMResponse 取回复文本**：`resp.completion_text`（property，源自 result_chain 或 _completion_text）。证据：官方 `plugin.md:1184-1252`。
**B 方案推荐**：把「画像+情景」注入 `req.system_prompt` 的 prepend 位置（画像是稳定的系统级背景，比塞进 user prompt 更合语义）。做成可配置。官方另有 `extra_user_content_parts.append(TextPart(...))` 方式可选。

### 3.4 钩子里必须包 try/except（降级安全）

Mnemosyne 每个钩子体都用 `try/except Exception` 包裹，出错只 `logger.error` 不抛出，保证记忆故障绝不阻断对话。证据：`main.py:530-622`（query_memory 整体 try）、`621-622`、`636-637`。**B 方案必须照做。**

### 3.5 如何从 event 确定性地取身份（关键，决定隔离正确性）

⚠️ 全部 AstrBot 官方源码实证（`astrbot/core/platform/astr_message_event.py`），方法签名钉死：

| 取什么 | 方法 | 返回值 | 官方源码出处 |
|---|---|---|---|
| 发送者 ID（QQ 号） | `event.get_sender_id()` | str（从 message_obj.sender.user_id） | `astr_message_event.py:202-207` |
| 发送者昵称 | `event.get_sender_name()` | str（可能空） | `astr_message_event.py:209-219` |
| 群号 | `event.get_group_id()` | str（私聊返 `""`） | `astr_message_event.py:194-196` |
| bot 自身 ID | `event.get_self_id()` | str（assistant 的 sender_id 用它） | `astr_message_event.py:198-200` |
| 会话 ID | `event.get_session_id()` / `event.unified_msg_origin` | str（会话唯一标识） | `astr_message_event.py:190-192`；文档 `plugin.md:750-751` |
| 纯文本消息 | `event.message_str` | str | 文档 `listen-message-event.md:35` |
| 是否群聊 | `event.get_group_id() != ""` | bool | 同 get_group_id |

→ **B 方案的身份映射全部基于以上官方方法**，不再依赖 Mnemosyne 的内部 helper。
- `user_id` = `event.get_sender_id()`（QQ 号，确定性）
- assistant `sender_id` = `event.get_self_id()`
- `session_id` = `event.unified_msg_origin`
- `is_group` = `event.get_group_id() != ""`
| 当前 provider（会话隔离） | `context.get_using_provider(umo=event.unified_msg_origin)` | `main.py:587-589` |

**B 方案铁律**：`user_id` 必须从 `event` 确定性取得（QQ 号），**绝不**像现有集成插件那样靠 LLM 传参或回退 `"default"`（见 4.2）。

## 第四部分：现有集成插件的反面教材（B 方案要避开的坑）

> `astrbot_plugin_everos_integration`（v1.1.0）是半成品，以下缺陷是 B 方案的「不要这样做」清单。

### 4.1 没有自动注入/归档钩子

`main.py` 里 grep 钩子注册，只有 `@filter.command_group("everos")` 和 `terminate`，
**完全没有 `on_llm_request` / `on_llm_response` / `on_astrbot_loaded`**。证据：`main.py:56,63,466,673`。
→ 它只是「被动工具库」：注册 3 个 LLM 工具（`everos_memorize/learn/recall`）让 Agent 主动调，不自动拦截对话流。
→ **B 方案的全部价值就在于补上这三个钩子。**

### 4.2 身份映射缺陷（会放大跨用户串线）

```python
# 证据：astrbot_plugin_everos_integration/tools/everos_tools.py:65, 223, 226-227
resolved_user_id = user_id or persona_name or "default"   # ← user_id 靠 LLM 传参，缺省落 "default"
...
candidate_uids = [resolved_user_id, "default"]            # ← 检索时轮询多个 uid
for uid in candidate_uids:
    result = await self._client.memory_search(..., user_id=uid, ...)
```
**后果**：多用户都可能落进 `"default"` 公共池；检索还主动把 `"default"` 池一起捞——
这等于把 EverOS 在引擎层做好的硬隔离（1.2）在集成层亲手破坏，**直接重蹈你审计文档 §15.9 的跨号串线**。
→ **B 方案铁律**：user_id 来自 event 的 QQ 号，确定性、单一、绝不轮询、绝不 default 兜底。

### 4.3 README 与代码不符

README v1.1.0 宣称有「对话积累模式」「双轨检索」，对应文件 `retrieval_hook.py` / `dialog_sync.py` 在代码里**不存在**。

### 4.4 可复用的部分（B 方案可借鉴）

## 第五部分：Mnemosyne 当前真实能力（说明「为什么不留在它身上」）

> 用于向用户/社区解释 B 方案的必要性。Mnemosyne v2.1.5 当前源码实证：

| 能力 | 现状 | 证据 |
|---|---|---|
| 会话级硬过滤 | ✅ 默认开（use_session_filtering=True） | `memory_operations.py:788-815` |
| 参与者过滤 | ⚠️ 可选软过滤 | `memory_operations.py:351-366` |
| 轻量记忆图谱 rerank | ✅ 有，但只是关键词共现，非语义 | `memory_operations.py:244-398` |
| 多向量库后端 | ✅ chroma/milvus/qdrant/weaviate | `memory_manager/vector_db/` |
| **持久用户画像** | ❌ 无 | 全仓库无 profile/bond-state 概念 |
| **巩固/去重/遗忘** | ❌ 无（只增不删） | grep 无 consolidat/dedup/forget |

**结论**：Mnemosyne 改进了会话隔离，但「持久画像」和「同质过饱和」这两个最痛的点，架构上仍无解——这正是 EverOS 引擎的强项，也是 B 方案的立足点。
