## 一、命名规范（防「中文键混英文」类病症）

来自 Mnemosyne 审计的命名教训，钉死以下规则：

| 项 | 规则 | 反例 |
|---|---|---|
| 结构标识符 | 全英文 snake_case，零中文键 | `用法=` `执行=`（禁） |
| 配置键 | snake_case 英文 | `everos_base_url`（对）；`记忆地址`（禁） |
| 常量 | 全大写 UPPER_SNAKE，集中在 constants.py | 散落硬编码字符串（禁） |
| 类名 | PascalCase | `EverOSClient` `Identity` |
| EverOS 字段名 | 严格照 api.md 原名 | `user_id`/`session_id`/`sender_id`/`app_id`/`project_id`/`memory_type` 一字不差 |
| memory_type 值 | 仅 `episode`/`profile`/`agent_case`/`agent_skill` | `user_profile`（禁，现有插件的 bug） |
| 注入标签 | 用插件专属、可配置 | 别和 Mnemosyne 的 `<Mnemosyne>` 撞 |
| 日志前缀 | 统一 `[everos]` | 便于在 AstrBot 日志里 grep |
| 代码注释/文档 | 中文（与用户语言一致），但标识符英文 | — |

## 二、可复用性要求（让开源社区用户能下载即用）

1. **零硬编码**：所有可变项（base_url、端口、scope、top_k、标签）走 `_conf_schema.json`。base_url 默认 `http://127.0.0.1:8000`，不写死 8765。
2. **everos_client.py 与 AstrBot 解耦**：client 模块不 import 任何 AstrBot 类型，输入输出纯 dict——这样它可被其它项目复用，也能脱离 AstrBot 单测。
3. **降级不崩**：EverOS 没装/没起，插件要能加载、要在 `/everos status` 里清楚报「EverOS 未连接」，而不是加载失败或每轮报错刷屏。
4. **配置有 hint**：`_conf_schema.json` 每项带 `description` + `hint`，照 `astrbot_plugin_everos_integration/_conf_schema.json` 的格式（已验证 AstrBot 能渲染）。
5. **首次使用引导**：README 顶部给「3 步跑起来」：装 EverOS → 填 base_url → 开插件。EverOS 部署指向其官方 QUICKSTART，不自己复述（避免文档漂移）。
6. **i18n 友好**：visibility 类目等用英文常量（PUBLIC/PRIVATE/TASTE/...），对应你审计文档 §15.6 的「开源复用性」原则。

## 三、可审计性要求（经得起 review 和迭代）

1. **关键决策留注释出处**：凡涉及「为什么这么做」的地方（尤其身份铁律、注入位置、memory_type 取值），注释里写明依据，引用 api.md 或反面教材行号。
2. **单一事实源**：常量只在 constants.py 定义一次；配置默认值只在 _conf_schema.json 定义一次，代码 `config.get(key, 默认)` 的默认值要与 schema 一致（或干脆全读 schema）。
3. **分层不串味**：identity 不调 EverOS；client 不碰 AstrBot；injection 不解析身份。模块职责见 `02-架构设计.md` 第二节。
4. **测试是回归防线**：身份隔离测试（test_identity.py 的串线回归）必须存在且通过——这是防 §15.9 复发的护栏，不是可选项。
5. **README 与代码强一致**：⚠️ 现有集成插件最大的信任问题是「README 宣称的功能代码里没有」。规约：**README 只描述代码里真实存在的功能**。发版前执行下方自检。

## 四、发版前自检清单（grep 可验证）

```bash
# 1. 无中文键名混入结构（应为空）
grep -rnE '"[一-龥]+"\s*[:=]' core/ main.py | grep -v description | grep -v hint

# 2. 无 "default" 兜底 user_id（应只在合法场景，人工核对每处）
grep -rn '"default"' core/identity.py

# 3. 无轮询多 uid 的反模式（应为空）
grep -rn 'candidate_uids\|\[.*user_id.*"default"\]' core/

# 4. memory_type 无 user_profile 误用（应为空）
grep -rn 'user_profile\|"foresight"' core/

# 5. base_url 不硬编码 8765（应为空或仅注释）
grep -rn '8765' core/ main.py

# 6. 每个钩子都有 try/except（人工核对 3 个钩子）
grep -n 'try:\|except' main.py

# 7. README 提到的每个功能，代码里都有对应实现（人工核对，重点查 README 的功能列表）
```

## 五、开源发布清单

| 项 | 要求 |
|---|---|
| 插件命名 | ⚠️ `name` 须 `astrbot_plugin_` 前缀、全小写、无空格、尽量短（官方 `plugin-new.md:21-25`） |
| `metadata.yaml` | name/display_name/desc/help/version/author/repo 齐全；字段照官方模板 `astrbot-plugin-template/metadata.yaml` |
| `_conf_schema.json` | 每项有 description+hint+type+default；type 取 `string/text/int/float/bool/object/list/dict` 之一（官方 `plugin-config.md:46`） |

| `README.md` | 功能列表（与代码一致）、3步快速开始、配置说明、与 Mnemosyne 的差异说明、EverOS 部署指引链接 |
| `迁移指南` | ⚠️ 从 Mnemosyne 迁来的用户必读：旧记忆迁移、prompt 契约重写（见第七节）、「换引擎=世界线变动」的预期管理 |
| `LICENSE` | ⚠️ 见下方许可证说明 |
| 版本号 | 从 v0.1.0 起；遵循语义化版本；CHANGELOG 记录每版真实改动 |
| 致谢 | README 注明基于 EverOS、参考了 Mnemosyne 的集成范式（尊重上游） |
| 发布市场 | 按官方 `plugin-publish.md` 流程；代码用 ruff 格式化 |

## 六、已知风险与诚实边界（写进 README，不藏）

| 风险 | 说明 | 缓解 |
|---|---|---|
| EverOS 基准成绩有第三方复现质疑 | Issue #73 / locomo-audit 仓库质疑 LoCoMo 复现 | README 不引用其 benchmark 数字；以「实测 profile 质量」为采用依据（任务 0） |
| EverOS 无主动遗忘 | 老 episode 永不删，长期可能膨胀 | 文档告知；靠聚类去重缓解；磁盘可定期归档 |
| EverOS 读写最终一致 | flush 后 search 可能 ~1s（高负载 10-15s）延迟 | 注入读历史记忆不受影响；归档后不立即依赖可检索 |
| EverOS 无内置鉴权 | 默认绑 127.0.0.1 | 本地部署安全；暴露公网需用户自加网关（README 警示） |
| visibility 非 EverOS 原生 | 隐私分层靠插件旁路 | MVP 做「群聊只给公开层」粗粒度；细粒度 v1.1 |
| EverOS 提取质量未经你的数据验证 | profile 能否扛深羁绊用户未知 | 任务 0 前置实测，不达标则不迁 |
| 每轮 search/归档增加延迟与 LLM 成本 | EverOS 提取要调 LLM | archive_strategy=auto 默认；注入可关 |

> 诚实边界原则直接来自你的审计文档：每个能力都标注「能做什么、不能做什么」，不夸大（对应 §15 反复出现的「边界·必须诚实」段）。

## 七、与现有 prompt 体系的迁移衔接（提醒，非本插件代码任务）

切到 EverOS 后，你现有的 Mnemosyne 配套 prompt 体系需配套调整，README 的迁移指南要如实告知：

1. **mnemosyne-archiver prompt 基本退役**：EverOS 用内部 LLM 做结构化提取（`extract_user_profile.py`），不再需要你的归档器 prompt 写叙述。归档器审计里的「QQ 号抄错」「记增量不记复读」等问题由 EverOS 的结构化提取 + 聚类去重接管。
2. **kurisu-persona 的 §11 运行时契约要重写**：注入格式从 `<Mnemosyne>` 标签变为本插件的 `<EverOS Memory>` 结构（画像段+情景段）。这正是你审计文档「教训 Y」——运行时契约必须对齐平台真实注入——的又一次应用。
3. **visibility taxonomy 平移**：你 §11.6 的 PUBLIC/PRIVATE 五类目规则，迁到本插件 `visibility.py`（v1.1）。
4. **旧记忆迁移**：Mnemosyne 在 Milvus 里的存量记忆需导出后按 EverOS 的 /add 格式重新灌入（一次性脚本）。这是「世界线变动」的代价，预期要管理好。

## 八、给迭代者的话

这个插件的立身之本是：**集成层只做集成，做得干净、确定、可降级；记忆智能交给 EverOS。** 
迭代时守住这条线——任何想在插件里重新实现「记忆提炼/遗忘/巩固」的冲动，先问一句「这是不是 EverOS 的职责」。
身份隔离的三条铁律（`02` 4.3）任何时候不能松动，那是用户用真实事故（§15.9）换来的教训。
