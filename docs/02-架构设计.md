# 架构设计 · 分层、数据流与关键决策

> 本文件定义 `astrbot_plugin_everos` 的内部架构。所有设计都对接 `01-实证依据.md` 的源码事实。
> 设计原则：**集成层只做集成**——身份映射、注入拼装、归档触发、降级安全；记忆智能全部交给 EverOS 引擎。

```

│   ├── injection.py           # 注入逻辑：检索结果 → 拼装记忆文本 → 改写 ProviderRequest
│   ├── archiving.py           # 归档逻辑：event+response → add/flush 触发策略
│   ├── visibility.py          # （可选 v1.1）visibility 旁路过滤

    └── test_injection.py      # 注入拼装单测
```

## 二、分层职责

```
┌─────────────────────────────────────────────────────────┐
│ main.py  (AstrBot Star)                                  │
│  - @register 注册插件                                     │
│  - 绑定 3 钩子 + terminate + /everos 命令组               │
│  - 每个钩子体：try/except 包裹 → 调 core 模块             │
└───────────────┬─────────────────────────────────────────┘
                │
   ┌────────────┼────────────┬──────────────┬──────────────┐
   ▼            ▼            ▼              ▼              ▼
identity.py  injection.py  archiving.py  visibility.py  commands/
  身份映射     注入拼装      归档触发       隐私过滤        管理命令
   │            │            │              │
   └────────────┴────────────┴──────────────┘
                │
                ▼
         everos_client.py  ──HTTP──▶  EverOS 服务 (用户单独部署)
```

| 模块 | 单一职责 | 绝不做 |
|---|---|---|
| `main.py` | 钩子/命令的绑定与异常兜底 | 不写业务逻辑 |
| `identity.py` | 把 event 确定性映射成 (user_id, session_id, app_id, project_id) | 不调 EverOS |
| `injection.py` | 调 search、把结果拼成文本、改写 req | 不解析身份（接收已解析的） |
| `archiving.py` | 决定何时 add、何时 flush，构造 messages | 不改 req |
| `everos_client.py` | 纯 HTTP 封装，输入输出都是 dict | 不碰 AstrBot 类型 |
| `visibility.py` | 按 visibility 规则过滤注入内容 | 不写 EverOS |

**依赖方向**：`main` → `core/*` → `everos_client` → HTTP。`everos_client` 不反向依赖任何 AstrBot 类型（便于脱离 AstrBot 单测）。

## 三、核心数据流

### 3.1 注入流（on_llm_request，每轮对话）

```
用户发消息
  │
  ▼
@filter.on_llm_request(event, req)
  │
  ├─ 1. 开关/前置检查（插件启用？该会话启用？EverOS 健康？）─── 否 → return（不阻断）
  │
  ├─ 2. identity.resolve(event) → user_id(QQ号), session_id, app_id, project_id
  │
  ├─ 3. query = 取 req.prompt（剥离群聊包装，参考 Mnemosyne ChatroomContextParser 思路）
  │
  ├─ 4. everos_client.search(
  │        user_id=user_id, query=query,
  │        include_profile=True,        ← 关键：恒取画像
  │        top_k=配置值, method="hybrid",
  │        app_id, project_id)
  │      → { profiles:[...], episodes:[...] }
  │
  ├─ 5.（可选）visibility.filter(results, is_group=event.get_group_id()!="")
  │        群聊只留 PUBLIC 层
  │
  ├─ 6. injection.build_text(profiles, episodes) → memory_block
  │        画像段（恒定）+ 情景段（按 score top-n）
  │
  └─ 7. injection.inject(req, memory_block, method, position)
           默认 req.system_prompt = memory_block + req.system_prompt
```

**注入文本结构建议**（区分「稳定画像」与「相关情景」，呼应你审计文档对"持久印象"的需求）：
```
<EverOS Memory>
【用户长期印象】          ← 来自 profiles[].profile_data，每轮恒注入
{summary / explicit_info / implicit_traits 渲染}

【相关情景记忆】          ← 来自 episodes[]，按 score 取 top-n
- [时间] {episode summary}
- ...
</EverOS Memory>
```
前后缀标签可配置（参考 `Mnemosyne` 的 `long_memory_prefix/suffix`，`memory_operations.py:995-998`），便于响应后从 contexts 里清理、避免污染历史。

> **渲染实现（已迭代）**：`_render_profile` 命中 EverOS 画像 schema 时渲染为整洁中文三段——
> `总体印象：…` / `显式信息：- [类别] 描述` / `隐含特质：- 特质（tags）：描述`，并**丢弃
> evidence/basis/profile_timestamp_ms 等溯源字段**（降噪省 token；溯源字段改由 `profile_quality`
> 校验消费）。非该 schema 的 profile_data 回退通用 `key：value` 平铺。
>
> **画像质量校验**：`core/profile_quality.py` 以确定性规则对单份 profile_data 做体检
> （必填完整性 / 结构有效性 / 缺证据=疑似臆测 / 重复特质 / 冗长），输出 `score+issues`，
> 经 `/epk quality` 抽查或离线验收脚本调用。**边界**：规则校验只保结构底线，无法识别
> "证据存在但推断错误"的语用误判（如把记忆测试误读为健忘），后者需 LLM judge，本期未做。

### 3.2 归档流（on_llm_response，每轮对话后）

```
LLM 产出回复
  │
  ▼
@filter.on_llm_response(event, resp)
  │
  ├─ 1. 开关/前置检查 ─── 否 → return
  │
  ├─ 2. identity.resolve(event) → user_id, session_id, app_id, project_id
  │
  ├─ 3. archiving.build_messages(event, resp) → [
  │        {sender_id: user_id(QQ号), role:"user", timestamp: now_ms, content: 用户原话},
  │        {sender_id: bot_id,        role:"assistant", timestamp: now_ms, content: resp 文本}
  │      ]
  │
  ├─ 4. everos_client.add(session_id, messages, app_id, project_id)
  │
  └─ 5. 按归档策略决定是否 flush（见 3.3）
```

**timestamp 必须是 Unix 毫秒**（`docs/api.md:208,231-238`）：`int(time.time() * 1000)`。
**sender_id 对 user 消息必须是 QQ 号**（它就是索引键，`docs/api.md:213-219`）；assistant 用稳定 bot 标识（如 `self_id` 或常量 `"assistant"`）。

### 3.3 归档触发策略（可配置，默认推荐）

EverOS 自己有「边界检测」会在 add 累积到一定程度自动 extract（`docs/api.md:466-469`）。集成层的 flush 策略：

| 策略 | 行为 | 适用 |
|---|---|---|
| `auto`（默认推荐） | 每轮只 add，依赖 EverOS 边界检测自动提取；会话静默 N 分钟后由定时任务 flush | 平衡，省 LLM 调用 |
| `every_turn` | 每轮 add 后立即 flush | 实时性高、LLM 成本高（每轮一次提取） |
| `manual` | 只 add，靠 `/everos flush` 手动触发 | 调试用 |

> 现有集成插件是「每条消息立即 add+flush」（`everos_tools.py:83-87`），LLM 成本最高。B 方案默认 `auto` 更经济。
> 注意最终一致性（`docs/api.md:79-90`）：flush 后立即 search 可能读不到，但注入读的是历史记忆、不依赖刚归档的内容，无影响。

## 四、身份映射设计（隔离正确性的命门）

### 4.1 映射规则

| AstrBot 来源 | EverOS 字段 | 规则 |
|---|---|---|
| 发送者 QQ 号（从 event 取） | `user_id` / `sender_id`(user) | **确定性必取**，取不到则跳过记忆（不 default 兜底） |
| `event.unified_msg_origin` | `session_id` | 直接用 umo 作 session（天然区分私聊/群/平台） |
| 人格名（persona） | `app_id` 后缀 | 在隔离白名单内的人格用独立 app_id，否则共享。参考 `config_manager.get_app_id_for` |
| 固定配置 | `project_id` | 默认 `"default"`，可配置 |
| bot 自身 id（self_id） | `sender_id`(assistant) | 稳定标识即可 |

### 4.2 私聊 vs 群聊

- **私聊**：`user_id` = 对方 QQ 号。直接、无歧义。
- **群聊**：每条消息的 `user_id` = 该消息发送者的 QQ 号（不是群号）。`session_id` = umo（含群标识）。
  这样同一群里不同人的记忆仍按各自 QQ 号隔离，但共享 session 上下文。
  群号可作为 `session_id` 的天然组成（umo 已含），不单独建 scope。

### 4.3 铁律（写进 identity.py 注释）

```
# 铁律 1：user_id 必须来自 event 的真实发送者 QQ 号，不接受外部传参。
# 铁律 2：取不到有效 user_id 时，返回 None，调用方跳过记忆操作——绝不回退 "default"。
# 铁律 3：检索只用单一 user_id，绝不轮询 [uid, "default"] —— 那会破坏 EverOS 的硬隔离。
# 反面教材：astrbot_plugin_everos_integration/tools/everos_tools.py:65,223,226-227
```

## 五、visibility 隐私分层（可选 v1.1，EverOS 不原生支持）

你审计文档 §11.6 的 PUBLIC/PRIVATE 五类目（TASTE/IDENTITY/ROUTINE/VULNERABILITY/BOND）是核心资产，但 EverOS schema 不支持自定义标签（`01-实证依据.md` 1.3）。两条旁路方案：

| 方案 | 做法 | 取舍 |
|---|---|---|
| **A. 注入层场景过滤（推荐 MVP+）** | 不改 EverOS 数据。在 injection 阶段，群聊时只注入 profile 的「公开摘要」字段、跳过 episode 细节；私聊才注入完整画像 | 简单、不依赖 EverOS 改造；粒度粗（按场景一刀切） |
| B. 旁路标签表 | 插件侧维护一张 `user_id → visibility 规则` 的本地表（SQLite/JSON），注入前按表过滤具体条目 | 粒度细、贴合五类目；需自己维护标签的打标与更新 |

**MVP 决策**：先做方案 A 的「群聊只给公开层」（直接对应你审计文档事故 Z 的封堵），方案 B 留作 v1.1 扩展点，在 `visibility.py` 预留接口。

## 六、降级与容错（绝不阻断对话）

| 故障 | 行为 | 依据 |
|---|---|---|
| EverOS 不可达 | 钩子 try/except 捕获，logger.warning，return（对话照常） | Mnemosyne `main.py:530-622` 范式 |
| search 超时 | httpx timeout（默认 30s，可配），超时按「无记忆」处理 | `everos_client.py:21` |
| search 返回空 | 注入空块或跳过注入 | — |
| add/flush 失败 | logger.error，不重试阻塞（可选后台重试队列，非 MVP） | — |
| user_id 解析失败 | 跳过本轮记忆操作 | 4.3 铁律 2 |

**原则**：记忆是增强项，不是关键路径。任何记忆故障都不能让用户收不到回复。

## 六之二、引擎连接形态（部署决定，配合 06）

部署形态已锁定：**EverOS 容器与 AstrBot 容器同处一个 docker 网络**（用户 AstrBot 在 Docker 里、接受一次手动 `docker compose up`）。对插件代码的影响：

- **base_url 默认用同网络服务名**：`http://everos:8000`，不是 `127.0.0.1:8000`。docker 网络内靠服务名解析，不碰宿主机 IP。
- **非容器/单机部署**：用户把 base_url 改成 `http://127.0.0.1:8000`（schema 的 hint 里注明）。
- **插件不调 docker**：纯 HTTP 客户端，引擎起没起只通过 `health()` 探测，不负责拉起/停止容器。
- **健康探测惰性化**：`initialize()` 探一次（失败只告警），每次钩子里再探，容器晚于 AstrBot 起来也能自愈连上。

详见 `06-Docker部署与开箱即用.md`。

## 八、与你的人格/归档器 prompt 体系的衔接（重要）

你现有体系（kurisu-persona + mnemosyne-archiver）是建立在 Mnemosyne 之上的。切到 EverOS 后：

1. **归档器 prompt 的角色变化**：EverOS 自己用内部 LLM 做 episode/profile 提取（`extract_user_profile.py`），**不再需要你的 mnemosyne-archiver prompt 来写归档叙述**。你的归档器审计里那些「LLM 手写 QQ 号抄错」「记增量不记复读」问题，由 EverOS 的结构化提取 + 聚类去重接管。
   → 但 EverOS 的提取质量需实测（它的 profile 长什么样、扛不扛得住茕霞那种深羁绊用户），这是迁移前必须验证的，列入 `03` 的前置任务。

2. **人格 prompt 的 §11 运行时契约要重写**：你审计文档教训 Y（运行时契约必须对齐平台真实注入）在这里重演——注入格式从 Mnemosyne 的 `<Mnemosyne>` 标签变成本插件的 `<EverOS Memory>` 结构，人格 prompt 里描述记忆注入的部分要同步改。

3. **visibility taxonomy 是你的资产**：EverOS 不原生支持，靠本插件 visibility.py 旁路实现（第五节）。你 §11.6 的五类目规则可以平移过来。

> 这些衔接不是本插件的代码任务，但新会话要在 README 和迁移指南里如实告知用户：**换引擎 = 一次世界线变动**，旧记忆迁移 + prompt 契约重写是配套工作。
